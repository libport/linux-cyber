# What's New In RHEL 9
The RHEL 9 edition of EX294 focuses on the Ansible features that differ from the earlier RHEL 8-based exam. It assumes the broader RHCE syllabus is already in place and concentrates on platform changes, collection-based content, stronger playbook design, and the execution environment used by `ansible-navigator`.

The course is treated as an update path rather than a full beginner course. It therefore concentrates on what changed, why those changes matter in practice, and how the working habits around Ansible should change with them. The recurring themes are explicit configuration, repeatable packaging, predictable recovery from failure, and safe testing inside containerised execution environments.

## Preparing the RHEL 9 lab
RHEL 9 includes `ansible-core` in the standard repositories. That simplifies a basic lab build because no extra repository is required for an initial install. In practice, `ansible-core` supplies the core engine and builtin content, but it leaves the system on Python 3.9. That combination works for many builtin modules and simple ad hoc checks such as `ansible localhost -m ping`, yet it can limit compatibility with newer collections and plugins.

A fuller lab setup enables the Red Hat Ansible Automation Platform repository through `subscription-manager`. That path installs the `ansible` package and brings in a newer Python interpreter, which the examples pair with Python 3.11. For exam preparation, the material aligns the exam with Automation Platform 2.2. The working lab, however, uses 2.5 to expose newer features and a more current Python stack. The key point is practical rather than promotional - the exam target and the practice environment may differ, and the operator needs to choose deliberately.

A sensible build sequence starts by confirming the host with `hostnamectl`, checking available repositories, and verifying the Ansible and Python versions after installation. The lab shown here runs on RHEL 9.5 with modest memory, around 2 GB, which is enough for command-line work and for container-based execution environments. The same approach can run on a physical host or in a virtual machine. The important requirement is not a specific hypervisor but a clean RHEL 9 control node with reliable package access and enough memory for container pulls and test runs.

A clone such as Rocky Linux 9 can also support the workflow. In that case, the practical equivalent is to enable EPEL and install Ansible from there, again with attention to Python compatibility. Regardless of the distribution, the first validation step stays simple - run `ansible --version`, then confirm that `localhost` responds to the `ping` module. That small test proves three things at once: Ansible is installed, Python is available to the control workflow, and the basic local execution path works.

This setup section also clarifies a subtle but important point. Installing the full `ansible` package does not replace Ansible with something unrelated to `ansible-core`. It uses the core engine but wraps it in a broader package choice that better suits modern collections, plugins, and execution environments. That distinction matters because the practical decision is not between two incompatible tools. It is between a minimal installation that is often sufficient and a more complete installation that reduces friction when newer content is required.
## Collections, namespaces and fully qualified names
Collections reshape how Ansible content is organised from version 2.10 onward. Earlier releases commonly used short module names such as `ping`, `service` or `file`. Newer releases still accept those short names in many cases, but the preferred form is the fully qualified collection name, or FQCN, such as `ansible.builtin.ping`.

That naming model solves a real scaling problem. As more contributors publish content, short names become harder to keep unique. The FQCN identifies the namespace, the collection, and the target module, so it reduces naming collisions and makes ownership clear. In `ansible.builtin.ping`, `ansible` is the namespace, `builtin` is the collection, and `ping` is the module. The same logic applies to routine modules such as `ansible.builtin.package`, `ansible.builtin.service`, and `ansible.builtin.file`.

The change also improves readability in long-lived automation. An ad hoc command benefits mainly from speed, so a short name is often acceptable at the shell. A playbook has a longer life and doubles as documentation, so the FQCN gives future readers better context. That same preference supports the broader Ansible habit of explicit configuration. Settings such as `gather_facts: false` state intent clearly and avoid hidden assumptions. Explicit `become` settings do the same for privilege escalation, especially when a role or task may later be reused in a different play.

`ansible-doc` provides the quickest way to confirm a module name and inspect its origin. It shows the FQCN, the module path, and the embedded documentation for builtin modules. That makes it easier to move from an older shorthand style to the clearer collection-based form without guesswork. The web documentation follows the same pattern and also highlights a version boundary. Older documentation for 2.9 still centres on the short name, while 2.10 and later present the collection-based form as the normal pattern.

Short names therefore remain a compatibility convenience, not the best long-term style. The safer habit for exam work and production code is to write playbooks with FQCNs, use short names only when shell speed matters, and treat the fully qualified form as part of the playbook's internal documentation. That change is small in syntax but large in maintainability.
## Roles, collections and Ansible Galaxy
Roles still group related tasks into reusable units. A simple web role can install `httpd`, ensure the service starts, and elevate privileges only where required. That reduces clutter in the playbook itself because the playbook can call the role rather than restating every package and service task. The role remains useful across multiple playbooks, so it should be written as durable, readable automation rather than a quick one-off script.

Traditional roles, however, sit loosely in the roles path and can collide by name. A role called `web` is easy to understand, but it is not inherently unique. Collections extend that idea by adding namespacing around roles and plugins. A collection avoids collisions by placing content inside a namespace and collection path, for example `namespace.collection.role`. Once installed, the playbook references the role through that full path, which makes origin and intent clearer.

The workflow is straightforward. Initialise a role when only a role is needed, or initialise a collection when the content should live under a namespace. Inside the collection, create roles in the `roles` directory, add tasks that use FQCNs, then build the collection into a tarball and install it with `ansible-galaxy collection install`. The playbook can then pull in the role by its namespaced identifier. This reduces duplication and helps teams grow their own internal content without trampling over external names.

Versioning matters because collections are packaged artefacts. When a task contains an error, such as using `enable` instead of `enabled` in a service definition, the fix should not overwrite history silently. The correct approach is to update the task, bump the version in `galaxy.yml`, rebuild the archive, and reinstall with `--upgrade`. That makes the dependency state visible and repeatable. It also turns a typo into a lesson about release discipline rather than a one-off edit on disk.

Collections can also include custom plugins. The examples add a Jinja filter plugin that converts text to uppercase. The code itself is less important than the pattern - place the filter in `plugins/filter`, expose it with a clear name, rebuild the collection, and call it in templating with its fully qualified filter path. That structure keeps custom extensions organised and avoids collisions with other plugins. It also demonstrates that collections are more than bundles of roles. They are the packaging format for reusable Ansible content.

Remote collections follow the same logic. Galaxy provides searchable community content, installation commands, documentation, and content listings. A collection such as `community.mysql` can be installed directly and then used to manage MySQL or MariaDB. The practical constraint is version compatibility. Newer collections may require a recent Ansible release, so keeping the control node current avoids unnecessary friction. The broader lesson is that Galaxy content should be treated as a dependency with documentation, version requirements, and maintenance implications, not just a command copied from a web page.
## Building robust playbooks
Robust playbooks depend on two habits - version control and deliberate error handling. A local Git repository gives each working state of a playbook a recoverable checkpoint. The examples create a repository, commit successive playbook versions, and branch each stable milestone before continuing development. Even without collaboration, that approach makes debugging faster because a known-good version is always available. It also makes regression testing possible when a newer edit breaks behaviour that previously worked.

The most basic error strategy is `ignore_errors: true`. It keeps execution moving when a task fails. That can be useful, but it is blunt. A better first step is to prevent predictable failures with logic. Gathering facts and using `when` clauses lets a playbook run Debian-specific package tasks only on Debian systems and Red Hat-specific package tasks only on Red Hat systems. That removes noise and produces clearer intent than allowing avoidable errors to occur. It also keeps reports cleaner because skipped tasks document platform branching more honestly than failed tasks that were expected to fail.

Fallback logic adds resilience when a failure is possible but acceptable. A task can `register` its result, ignore the immediate error, and then trigger a replacement task when `result.failed` is true. The examples use that pattern to attempt a file copy for a web page and then fall back to inline content when the source file is missing. The playbook still converges on a usable result. That is the real design goal - not merely to keep running, but to finish in a known state.

Retries help when a service is slow rather than broken. A `uri` check against `http://localhost` can `register` the response, retry several times, delay between attempts, and finish only when the status code reaches `200`. That is a cleaner response to start-up latency than an immediate hard failure. It also captures an important operational truth. Many failures are timing issues, not logical errors, so a short retry loop can turn brittle automation into stable automation without hiding genuine faults.

For more structured recovery, Ansible supports `block`, `rescue`, and `always`. A `block` groups the main tasks. `rescue` handles failure paths. `always` runs regardless of success or failure. That pattern mirrors exception handling in programming and fits playbooks that need both recovery actions and final validation. It can replace a scattered mix of conditional checks with a more coherent flow - attempt the normal path, recover if required, then execute the final common tasks.

The playbook examples also reinforce several design habits that matter beyond error handling. Use `become` at the narrowest reasonable scope. Name tasks clearly so lint tools and later readers can understand them. Prefer idempotent modules over raw shell commands. Treat stable versions in Git as checkpoints. Together, those habits produce playbooks that are easier to read, safer to rerun, and simpler to recover when the environment does not behave as expected.
## Preferring playbooks to shell scripts
Shell scripts still work, but Ansible modules usually express intent more clearly and require less defensive code. A simple Bash script that checks `/etc/passwd`, adds a user, and sets a password handles only one narrow path. It needs extra logic for missing arguments, different operating systems, user modification, and deletion. The equivalent Ansible task can manage presence or absence, apply a password hash, and remain idempotent across repeated runs.

The `ansible.builtin.user` module shows this advantage clearly. It can create or modify a user with a declarative state, set a hashed password, and limit password changes with `update_password: on_create`. Combined with targeted privilege escalation, that gives safer and more maintainable automation than a shell script that pipes plaintext credentials into system tools. Re-running the playbook confirms the result because the desired state already exists and no unnecessary change occurs.

This shift is really a change in mindset. When a task needs automation, the first question should be whether an Ansible module already captures the desired state. If the answer is yes, the module usually brings built-in idempotence, clearer parameters, and better portability than a handcrafted script. That does not make shell scripting obsolete, but it moves shell usage towards edge cases rather than default practice.

The same mindset applies when a vendor publishes shell commands. External documentation often presents a command-line recipe for convenience. In an Ansible workflow, the better response is to translate that recipe into a task. The Puppet repository example takes a raw RPM installation command and turns it into `ansible.builtin.yum_repository` with a repository name, description, `baseurl`, GPG key, architecture-aware path, and explicit state. Once written as a task, the repository configuration becomes readable, repeatable, and idempotent.

This approach also exposes missing dependencies earlier. When password hashing produced a warning, the fix was to install the required Python package that supports the chosen hash method. The lesson is practical - treat warnings as maintenance work, not background noise, so automation stays ahead of deprecations and dependency breaks. That same discipline applies to vendor repositories, external keys, and architecture-specific URLs. Putting those details into playbooks makes them reviewable and testable.

In short, playbooks replace imperative command sequences with declared outcomes. That is why the Ansible version of the task scales better than the shell version. It handles more lifecycle states, records intent more clearly, and supports repeated execution without custom guard logic at every step.
## Using Ansible Navigator and execution environments
`ansible-navigator` provides a controlled execution environment for testing and running playbooks. After installation, it uses Podman to pull an execution environment image from `registry.redhat.io`. Access requires authentication with a Red Hat service account, so a successful `podman login` is part of the setup, not an optional extra. Once the image is present, Navigator can launch an interactive menu and run Ansible tools inside a container rather than on the host system directly.

That isolation is useful in several ways. The container becomes the effective `localhost`, so test runs do not change the administrator's workstation or the target estate by accident. Inventory inspection, linting, and playbook execution all happen in a disposable runtime. The image persists locally after download, while containers are started and cleaned up as needed. This makes Navigator a practical lab environment even when the operator has only one machine available.

Inventory browsing is one example. A small test inventory with Linux and Windows groups can be loaded into Navigator and explored interactively. The hosts do not need to exist because the exercise focuses on validating structure, group membership, and visibility. That makes Navigator a safe inspection tool before any real execution occurs. It also reinforces how inventory is interpreted by Ansible, which helps when debugging group layout and host targeting.

Linting is another strong use case. `ansible-navigator lint` reports issues such as poor task naming, trailing whitespace, unnecessary blank lines, and unsafe file creation without explicit permissions. Those are not cosmetic details. They improve readability, reduce ambiguity, and push playbooks towards a maintainable standard before they are shared or reused. A playbook that runs is not automatically a good playbook. Linting closes that gap.

Navigator also supports interactive and non-interactive execution. It can run a playbook from the menu, open the play and task results for inspection, or print standard output directly with `--mode stdout`. Adding verbosity exposes more detail when debugging. Each run can also leave an artefact file in JSON format, which preserves the execution record and helps diagnose failures after the fact. Those artefacts matter because they provide a durable trace even after the container itself has exited.

The broader value of Navigator is consistency. It packages the runtime, standardises testing, and reduces the mismatch between one operator's laptop and another operator's control node. For EX294 on RHEL 9, that matters because the execution environment is part of the platform expectation, not an optional extra wrapped around Ansible. Together, execution environments, linting, inventory tools, and artefacts turn Navigator into a practical test harness for the RHEL 9 workflow.
