# Chapter 8: Technical Footprinting
## Scope and lawful boundaries
Technical footprinting collects public technical information about an organisation's web presence, domain names, DNS records, certificates, host providers, network ranges, exposed documents and internet-facing services. OSINT work relies on lawful, publicly available information, but public access does not remove privacy, copyright, contractual or cybercrime obligations. The investigator defines a written scope, records the source of each finding, and avoids testing that changes, bypasses, degrades or exploits a system without explicit authority.

Passive reconnaissance uses third-party records, cached pages, search indexes, certificate transparency logs, public DNS data, leak-free document metadata and reputation services. Direct browsing of a target website creates HTTP requests and appears in server logs, so it is low-interaction collection rather than pure passive collection. Mirroring a site, brute-forcing subdomains, route tracing, directory discovery, vulnerability scanning, password testing and social engineering are active or intrusive activities. They require authorisation and rate limits. Investigators should also avoid collecting personal information that has no clear security purpose. When personal data is relevant, the report should minimise exposure, protect copies and explain why the information matters to the assessment.
## Website inspection
A public website often reveals addresses, business units, staff names, job adverts, partners, support channels, technologies, policies, downloadable files and change history. Job adverts can disclose vendors, operating systems, cloud platforms, security tools, programming languages and branch-specific duties. These details help map capability and risk, but investigators should verify them against current sources because job advertisements can lag behind production architecture.

HTML, CSS, JavaScript and HTTP headers can reveal frameworks, libraries, analytics tags, content delivery networks, mail providers, security headers and configuration mistakes. Comments in source files may disclose development notes, staging links, usernames, internal hostnames or outsourced suppliers. Public files such as PDFs, spreadsheets, images and office documents can expose metadata including author names, usernames, software versions, template paths, timestamps and internal directory structures.

`robots.txt` gives crawler instructions. It does not protect sensitive content, and a `Disallow` entry may still reveal paths worth defending. Archive services can expose older pages and removed files, while backlinks and extracted links can reveal related domains, subdomains, APIs, scripts, embeds and third-party services. Website change monitoring can surface new vacancies, documents, portals and support pages, but collection must respect terms of use, rate limits and consent.
## Public records and technology discovery
Technology profiling combines several weak signals rather than relying on one tool. Services such as Wappalyzer, BuiltWith, Netcraft, Security Headers, URLScan, Censys and certificate transparency search can identify visible technologies, trackers, certificates, hosting changes, mail records and security headers. Their findings should be treated as indicators, not proof, because caching, front-end obfuscation, shared infrastructure and defensive configuration can hide or misattribute back-end systems.

The investigator uses vulnerability databases and vendor advisories to contextualise observed technologies. Public exploit sites do not identify private zero-days. They catalogue disclosed vulnerabilities, proof-of-concept exploit code, misconfigurations and historic advisories. The safer conclusion is a risk hypothesis, not a claim of compromise. Default credential lists can support defensive audits, but attempts to log in to someone else's devices cross into unauthorised access.
## Domain, DNS and network footprinting
Domain registration lookup has shifted from classic WHOIS toward RDAP for generic top-level domains. Modern registration records commonly show registrar, creation and expiry dates, nameservers, domain status, abuse contacts and selected registrant information. Privacy services and data-protection rules often redact personal details. Regional Internet registries, including APNIC, ARIN, RIPE NCC, AFRINIC and LACNIC, provide authoritative allocation data for IP address space and autonomous system numbers.

Subdomain discovery draws on search engines, certificate transparency logs, DNS databases, VirusTotal, DNSDumpster and other public repositories. Development, staging, support, mail, VPN, SSO, API and content subdomains can disclose architecture and exposed attack surface. Some tools mix passive collection with active brute-force or probing, so the method matters as much as the result.

DNS records describe how a domain uses internet services. Common records include A for IPv4, AAAA for IPv6, CNAME for aliases, MX for mail exchangers, NS for nameservers and TXT for text-based policies such as SPF, DKIM and domain verification. Queries to authoritative nameservers can confirm current records. Responses from recursive resolvers may be cached or non-authoritative.

Route tracing and port scanning interact with infrastructure and should not be labelled passive. Route tracing sends packets that may be filtered by routers or firewalls, and the observed path can vary between attempts. IP geolocation, reverse DNS, reverse IP hosting, ASN lookups, BGP data and blocklists can add context about hosting providers, shared infrastructure, routing and reputation, but they rarely prove a user's physical location or intent.
## Reporting
A technical footprinting report should separate observed facts, inferred relationships and risk judgements. It should capture source, date, method, confidence and limitations for each finding. Useful outcomes include a domain and subdomain inventory, certificate and DNS review, exposed metadata review, technology and dependency profile, risky public documents, outdated services, weak headers, third-party exposure and change-monitoring priorities. Good footprinting reduces surprise, supports defensive remediation and stays inside the bounds of law, authorisation and professional ethics. The report should recommend practical fixes, including removing sensitive documents, hardening headers, reducing metadata leakage, closing forgotten staging assets, correcting DNS records, improving certificate hygiene and documenting ownership of public-facing services.
