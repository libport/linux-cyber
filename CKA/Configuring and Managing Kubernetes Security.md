# Configuring and Managing Kubernetes Security
## Access control
Kubernetes security starts with the API server because every cluster action flows through it. Role-based access control defines who can do what. A Role grants verbs such as get, list, create and delete against resources such as pods, services and secrets inside one namespace. A RoleBinding grants that Role to users, groups or service accounts. A ClusterRole can define permissions for cluster-scoped resources or provide reusable permission templates for namespaces. A ClusterRoleBinding grants ClusterRole permissions across the cluster.

Kubernetes evaluates each request by authenticating the caller, collecting all matching RoleBindings and ClusterRoleBindings, and allowing the request only when a rule grants the requested verb on the requested resource. No matching permission means no access. This model makes permissions additive and explicit, which suits application-owned YAML and reviewable change control.

Cluster-admin access should remain a controlled break-glass permission, not a daily default. Binding cluster-admin to the default service account gives every pod that omits a serviceAccountName full cluster authority. A compromised container can then read secrets, delete workloads and affect system namespaces. Dedicated service accounts and least-privilege RoleBindings contain each workload to the resources it needs. A pod management application might delete pods in its own namespace while receiving read-only access in kube-system, with no permissions elsewhere.

Applications that do not call the Kubernetes API should not receive API tokens. Setting automountServiceAccountToken to false on the service account or pod removes a common credential exposure path. Current Kubernetes projected service account tokens are bound to pods and are time-limited by default, so operators should avoid legacy long-lived token patterns and remove unneeded default token mounts.

Human access needs separate treatment. Kubernetes recognises users but does not store User objects. External systems authenticate users through certificates, bearer tokens, authenticating proxies or OpenID Connect. Static kubeconfig files with embedded certificates or tokens create a password-file risk, especially when administrators switch between test and production clusters. Managed identity integration, such as Microsoft Entra ID for AKS, moves authentication to an identity provider that can enforce multi-factor authentication, conditional access and short-lived tokens. Kubernetes RBAC should then authorise the authenticated identity and group claims. Local administrator credentials should be disabled where the platform supports it, with an audited break-glass process retained outside routine access.

Authentication and authorisation should remain separate design concerns. The identity provider proves who the caller is, while Kubernetes decides what that caller can do. This separation lets an organisation apply central account lifecycle controls, password policy, multi-factor authentication and device rules without placing those policies inside every cluster. It also lets application teams model permissions close to the workloads they own.
## Deployment security
Every container image carries a software supply chain. A simple application image can include an operating system base, language runtime, web framework and many transitive dependencies. Each component follows its own release schedule, and any emergency security patch can force compatibility work. A clean scan at release time can become stale when a new CVE appears after deployment. Security teams need continuous image scanning in the cluster, not only scanning in CI/CD.

Trivy Operator can scan running workloads and publish VulnerabilityReport resources through the Kubernetes API. These reports support audit trails, ticket creation and policy decisions. They also create an inventory of vulnerable images across namespaces, which helps teams prioritise critical fixes and prove audit coverage.

Admission control should treat vulnerability data with care. Blocking new pods for an already-running vulnerable image can break scaling or recovery before the team has patched the issue. Audit mode often suits CVE checks, while hard enforcement suits stable configuration requirements. Critical CVEs can trigger alerts and tickets, while deployment gates can stop images from unapproved registries or workloads that ignore mandatory security contexts.

Kyverno provides Kubernetes-native policy enforcement. Enforce policies can block unsafe workloads before they run. Useful baseline rules include requiring approved registries, preventing root containers, denying privileged mode, dropping dangerous Linux capabilities and limiting writable filesystems where applications support it. Audit policies can record exceptions and guide future hardening without disrupting production.

Good policy design improves delivery rather than slowing it. Clear admission messages tell developers what failed and how to fix it. Policy reports show which teams need exemptions, where base images lag and which controls need adjustment. Automation moves repeated security judgement into the platform. Developers then receive consistent guardrails instead of relying on memory under delivery pressure.
## Runtime containment
Runtime controls reduce damage after compromise. Kubernetes networking allows broad pod-to-pod communication unless the cluster networking implementation enforces NetworkPolicy. A default-deny policy in each namespace creates an allow-list model for ingress and egress. Teams then add explicit rules for required paths.

A typical three-tier application should permit web pods to reach API pods, API pods to reach database pods and all required pods to reach DNS. The web tier should not connect directly to the database. Workloads should not reach the internet unless they need that path. Network policies are additive, use label selectors and apply at runtime, so operators can change segmentation without redeploying applications. The cluster must use a network plugin that supports NetworkPolicy for these rules to work.

This segmentation turns a compromised pod from a cluster-wide jump box into a contained workload. An attacker who controls the web tier should not call the database, scan system namespaces, download tools or exfiltrate data to external servers. Logs and runtime detection tools such as Falco can then add visibility when a process behaves suspiciously, while Kubernetes audit logs record API activity for investigation.

Secrets need the same containment approach. Kubernetes Secret values are base64 encoded, not encrypted by that encoding. By default, Secret objects are stored unencrypted in etcd unless the cluster configures encryption at rest. Anyone with read access to secrets can retrieve sensitive values, and environment variables expose credentials to anyone who can inspect or exec into the container. Source-controlled Helm values and plain Kubernetes Secret manifests create another avoidable exposure path.

External secret stores reduce this risk. Azure Key Vault, AWS Secrets Manager, Google Secret Manager and HashiCorp Vault keep credentials outside Kubernetes and add central access control, versioning, rotation and audit logging. The Secrets Store CSI Driver mounts selected secrets, keys or certificates into pods as volumes. Workload identity links a Kubernetes service account to a cloud identity, so pods can fetch secrets without static cloud credentials in manifests. A pod can still read its mounted secret, but the secret no longer needs to live in source control, environment variables or Kubernetes Secret objects.

Runtime hardening should proceed incrementally. Some controls, such as non-root execution and disallowing privilege escalation, suit most modern applications. Others, such as read-only root filesystems or strict seccomp profiles, can expose hidden assumptions in older images. Operators should apply each control, test the application path and then enforce the rule through policy once the workload proves compatible.
## Defence in depth
Defence in depth combines controls rather than relying on one barrier. RBAC and identity provider integration restrict who can use the API. Service account hardening restricts what workloads can do. Image scanning and admission policies control what can run. NetworkPolicy restricts movement between components. External secret stores reduce credential exposure. Pod security settings, including seccompProfile, runAsNonRoot, allowPrivilegeEscalation false, readOnlyRootFilesystem and dropped capabilities, add container-level containment after careful testing.

These layers make attacks harder and noisier. A stolen kubeconfig should not grant lasting access. A compromised pod should not read unrelated secrets, delete critical workloads, call the API without need, contact every namespace or exfiltrate data freely. New vulnerabilities and attack techniques will keep appearing, so Kubernetes security needs continuous scanning, enforced baselines, auditable exceptions and regular review.
